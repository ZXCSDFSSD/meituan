数据分析逻辑 — 原始数据 → 数据库 → API
菜品销售明细（品项销售明细）
  • 唯一含机构编码（→ stores 表，必须优先处理）
  • 品项名、品类大类/小类、数量、金额、收入、优惠
  • 点菜时间 → 提取 hour(0-23) → timeslot_summary
  • 下单时间所属餐段 → meal_period 字段
  • 订单来源 → resolveChannel → channel 字段
  • → 写入：order_dishes / item_sales_summary / timeslot_summary
全渠道订单明细
  • 无机构编码，门店名称查 stores 表获取 store_id
  • 订单号、订单金额、顾客实付、订单收入、优惠
  • 订单来源 + 用餐方式 → resolveChannel → channel
  • 创建时间 → business_date（营业日期）
  • → 写入：orders 表
  • → 聚合：sales_summary（日度）/ monthly_summary（月度）
  • 两表各含渠道拆分行（all / 堂食 / 外卖各渠道）
收款明细
  • 无机构编码，门店名称查 stores 表获取 store_id
  • 结账方式（微信/支付宝/现金等）
  • 支付金额、收款金额、手续费、到账金额
  • 业务小类（区分堂食/外卖收款）
  • → 写入：payment_method_summary（按门店+日期+方式聚合）
渠道解析规则（resolveChannel）
  • 订单来源含"美团" → 美团外卖
  • 订单来源含"饿了么"/"淘宝"/"闪购" → 饿了么（同一平台）
  • 订单来源含"京东" → 京东秒送
  • 用餐方式="外卖"/"外带" → 外卖（未知子平台）
  • 其余（POS/收银/店内）→ 堂食
  • summary 表额外存 channel='all' 汇总行
  • 前端：全渠道=all / 外卖=美团+饿了么+京东之和
时段统计逻辑（timeslot_summary）
  • 数据来源：菜品销售明细的"点菜时间"字段
  • 格式：2026/02/26 20:04:33
  • 提取：正则 \s(\d{1,2}): → hour 0-23
  • 聚合维度：store_id + date + hour + channel
  • 指标：TC（不同order_id数）+ total_amount + total_income
  • 额外存 channel='all' 汇总行
  • 前端展示：补全0-23所有时段，无数据填0
store_id 映射关键链路
  • 菜品销售明细 → 含"机构编码" → 写入/更新 stores 表
  • 全渠道订单明细 → 按"门店名称"查 stores 表 → 取 store_id
  • 收款明细 → 按"门店"查 stores 表 → 取 store_id
  • 处理必须按序：菜品 → 订单 → 收款
  • 未匹配门店名 → store_id 为 null（不报错，正常写入）
  • 新店首次导入后 store_id 自动写入 stores 表
品类标准化（category_mapping）
  • 各门店品类名称不统一（~98 种原始名，含 emoji 变体）
  • category_mapping 表：raw_name + store_id → standard_name
  • NULL store_id = 全局映射；非NULL = 门店级覆盖
  • item_sales_summary 目前存储原始品类名
  • API /category-sales 查询时 JOIN category_mapping → 标准化名称
  • ✅ 已完成：99 条全局映射，归并为 9 大标准品类
