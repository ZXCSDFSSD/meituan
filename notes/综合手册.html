一、SQL 视图评估——要不要建？

接口规范 Section 2 建议了两个视图。结论：当前不需要建，留作可选优化。

v_annual_compare（同比环比年度视图）
  • 目的：方便一次 SELECT 得到两年数据，不用写两次 JOIN
  • 当前状态：/monthly-compare API 已通过 JS 的 thisYearMonths / lastYearMonths 数组处理，前端 S2透视表正常显示
  • 结论：暂不建——API 工作正常，视图只是方便后续直接写 SQL 查询，目前不影响功能
  • 未来触发时机：需要导出报表到 Excel 时，直接 SQLite 查询更方便
v_category_monthly（品类月度透视视图）
  • 目的：把 item_sales_summary 和 category_mapping 的 LEFT JOIN 封装成视图
  • 当前状态：/category-sales API 内部已做这个 JOIN，品类横条图正常显示
  • 结论：暂不建——功能已实现，视图只减少重复 SQL 书写
  • 未来触发时机：如果后续要在 analytics.js 外部多处复用品类查询

💡建视图的正确时机：当同一个复杂 JOIN 在代码中出现 3 次以上，或需要直接用 SQLite 工具（如 DB Browser）导出数据时，再建视图。过早优化会增加维护负担。

二、架构思维总结——如何一步步把需求变成代码

需求拆解过程

原始需求是：11个 Excel Sheet → Dashboard 可视化。拆解过程如下：

// 分析师给出 Excel → 我需要反向推导数据库结构 步骤1：识别"事实表"和"维度表" - 事实：订单（orders）、品项明细（order_dishes）、收款（payment_method_summary） - 维度：门店（stores）、时段（timeslot_summary）、品类（category_mapping） 步骤2：识别数据源 → 每张表从哪个 Excel 来 - 全渠道订单明细 → orders + sales_summary + monthly_summary - 菜品销售明细 → order_dishes + item_sales_summary + timeslot_summary - 收款明细 → payment_method_summary 步骤3：识别"连接键"的唯一来源 - store_id 唯一来源是菜品销售明细（机构编码列） - 其他两个文件只有门店名称，必须先处理菜品才能 JOIN 步骤4：确定处理顺序 - 菜品销售明细 → 全渠道订单明细 → 收款明细 - 这个顺序不能打乱

关键技术决策（每个决策的理由）

决策选择理由放弃的方案
数据库 SQLite 单机部署、无需网络、嵌入式、Python/Node 直连、文件可备份 PostgreSQL（太重）、MySQL（需安装服务）
数据预聚合 monthly_summary 表（导入时计算） API 响应变为 O(1) SELECT；代价是导入时间更长（可接受） API 实时计算（每次请求跑聚合，慢）
前端状态 Zustand 5 + persist 状态只有月份/门店/渠道/主题，不需要 Redux 的复杂中间件 Redux Toolkit（过度工程）、React Context（性能差）
页面数量 3页（总览/趋势/单店） 分析后发现渠道/品类/时段数据量适合内嵌，不需要独立页 7页（规范中规划，实际简化）
渠道标准化 淘宝闪购=饿了么 业务知识：饿了么被阿里收购后技术合并，闪购=饿了么 单独保留"淘宝"（会分裂统计）
品类标准化 category_mapping 映射表 98种原始名→9大标准品类，用数据库映射比代码 switch/case 更灵活可维护 代码硬编码 regexp（难以扩展）
S2 透视表 命令式 API（new PivotSheet） @antv/s2 v2 是命令式库，不是 React 组件；必须配合 useRef + useEffect 无可替代方案（库本身的限制）
代码注释 三层注释（What/Why/How） 目标读者是只懂 JS 的开发者；注释要教学而不只是描述 只写 JSDoc 类型注释（信息量不足）

调试思路回顾（遇到错误怎么想）

🐛 TS2698 TypeScript 泛型错误
  • 错误：Spread types may only be created from object types
  • 原因：对泛型参数 T 做展开 {...props as T}，TS 严格模式下不知道 T 是对象类型
  • 思路：不要让类型系统"猜"，改为使用具体类型引用（Record<string, unknown> 或具体接口）
  • 教训：TypeScript 错误通常指向"你给的信息不够"，解决方法是"给更多具体类型信息"
🐛 API 启动失败（数据库未初始化）
  • 错误：直接运行 node src/api/index.js → 数据库不存在
  • 原因：api/index.js 只导出 startServer 函数，没有初始化 DB
  • 思路：溯源启动流程 → main.js 先调用 storage.initDatabase() 再调用 startServer()
  • 教训：模块化代码中，顺序依赖必须通过入口文件控制
🐛 品类显示碎片化（~98种）
  • 现象：/category-sales 返回几十个品类，图表无法看
  • 根因：各门店对同一品类的命名不一致（emoji/空格/别名变体）
  • 解决：建 category_mapping 表，99条映射 → 9大标准品类；API 查询时 JOIN 标准化
  • 教训：用户数据从来不干净，标准化层是必须的
🐛 S2 实例 React 状态冲突
  • 现象：S2 在 React 状态更新时报错/重复渲染
  • 原因：S2 是命令式库，有自己的生命周期；React 重渲染时不自动 destroy
  • 解决：用 s2Ref = useRef(null) 保存实例；useEffect 的 cleanup 函数里调用 s2.destroy()
  • 教训：命令式库接入 React 的标准模式：useRef + useEffect + cleanup

问题说明影响解决/绕开

数据导入注意事项

⚠️Fix2 品类 "--" 修复需要重新导入历史数据才生效！如果只新增导入，历史数据中的 dish_category = "--" 不会自动更新。需要先清空 item_sales_summary 相关数据，再重新 node main.js --import。

  • 导入顺序不能变：菜品销售明细 → 全渠道订单明细 → 收款明细（pipeline.js 中已固定）
  • 跳过已处理文件：importAll 会检查文件是否已导入（processed_files 表），同一文件不会重复导入
  • 月份提取：文件名格式 XXX_2025-01-01_2025-01-31.xlsx，从开始日期提取月份
  • 合计行过滤：Excel 末尾的"合计"行在解析时过滤掉（storeName === '合计' || '- -'）
  • 表头偏移：美团 Excel 前 2 行是标题和筛选条件，xlsx.utils.sheet_to_json({range:2}) 跳过

前端开发注意事项

  • S2 必须调用 destroy()：否则内存泄漏，useEffect cleanup 里调用
  • ECharts notMerge=true：当 option 完全变化时用 notMerge prop，否则旧数据会和新数据合并出错
  • Zustand partialize：不要把 loading/error 状态持久化到 localStorage，只持久化月份/主题等用户偏好
  • TypeScript 泛型展开：不要对泛型参数 T 直接做 {...obj as T}，改用具体类型或 Object.assign
  • 可选链防崩：API 返回数据前 data?.months ?? []，避免 map/filter 在 null 上报错
  • Windows 无 /dev/stdin：测试时不能用 curl | node -e "readFileSync('/dev/stdin')"，直接用 curl -s URL 查看
